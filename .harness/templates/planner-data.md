## 입력
- **미션**: {objective}
- **제약**: {constraints}
- **도메인 힌트**: etl_stages

# 프로젝트 사양: {project_name}

## 1. 목표
[사용자가 최종적으로 경험하는 것을 1~3문장으로]

## 2. 기능 목록
- **F-001**: [사용자 관점 기능 설명]
  - 완료 조건: [관찰/측정 가능한 조건]
  - 의존성: [없음 | F-xxx]

## 3. 스프린트 배치

## 데이터/크롤러 도메인 분해 규칙

기능 분해 시 ETL 단계를 따르세요:

1. **Source Definition**: 대상 사이트/API 정의
   - "사용자가 크롤링 대상 URL 목록을 입력할 수 있다"
   - "API 엔드포인트에서 데이터를 요청할 수 있다"

2. **Extraction**: 데이터 수집 로직
   - "대상 페이지에서 제목, 날짜, 본문을 추출한다"
   - "API 응답에서 필요한 필드를 추출한다"

3. **Transformation**: 데이터 정제/변환
   - "추출된 데이터에서 중복을 제거한다"
   - "날짜 형식을 ISO 8601로 통일한다"
   - "결측값을 처리한다"

4. **Loading**: 저장소 적재
   - "정제된 데이터를 지정된 형식으로 저장한다"
   - "저장 시 기존 데이터와 병합한다"

5. **Scheduling & Recovery**: 실행 주기 및 오류 복구
   - "주기적으로 자동 실행된다"
   - "실패 시 재시도 후 알림을 발송한다"

스프린트 배치:
- Sprint 1: Source Definition + Extraction 기본
- Sprint 2: Transformation
- Sprint 3: Loading
- Sprint 4: Error Recovery + Resilience
- Sprint 5: Scheduling (선택)

## 규칙
1. 기능 설명에 기술 용어를 사용하지 마세요.
2. 완료 조건은 반드시 검증 가능해야 합니다.
3. 한 스프린트에 3개 이상의 기능을 넣지 마세요.
4. 의존성이 없는 기능을 가능한 앞 스프린트에 배치하세요.
5. 기술 스택, 라이브러리, 아키텍처를 언급하지 마세요.
6. spec.md는 "검증 가능한 문장"으로만 구성하세요.
